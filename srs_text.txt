1. Pendahuluan 
1.1 Tujuan Penulisan Dokumen 
Dokumen Software Requirements Specification (SRS) ini bertujuan untuk menjelaskan secara 
rinci kebutuhan perangkat lunak pada Sistem Informasi Bengkel Motor Fatia Kamal. Dokumen 
Software Requirements Specification (SRS) ini disusun dengan tujuan sebagai berikut: 
1. Menyediakan pedoman bagi penguji (tester) dalam menyusun skenario dan prosedur 
pengujian sistem berdasarkan kebutuhan yang telah ditentukan. 
2. Menjelaskan kebutuhan sistem yang akan dikembangkan pada Sistem Informasi Bengkel 
Motor Fatia Kamal secara jelas dan terstruktur. 
3. Memberikan pemahaman yang sama bagi semua pihak terkait, termasuk pemilik bengkel, 
analis sistem, dan tim pengembang mengenai fungsi dan batasan sistem. 
4. Menjadi acuan bagi pengembang (developer) dalam proses perancangan, pengkodean, 
dan implementasi sistem agar sesuai dengan kebutuhan pengguna. 
1.2 Audien yang Dituju dan Pembaca yang Disarankan 
 Dokumen ini dituju untuk beberapa pihak yang terlibat dalam proses perencanaan, 
pengembangan, dan penggunaan Sistem Informasi Bengkel motor Fatia Kamal. Adapun audien yang 
dituju adalah: 
1. Pemilik Bengkel Motor Fatia Kamal, sebagai pihak utama yang membutuhkan sistem 
untuk membantu kegiatan operasional bengkel. 
2. Tim Pengembang (Developer), sebagai panduan dalam merancang dan membangun 
sistem sesuai kebutuhan pengguna. 
3. Analis Sistem, untuk memahami alur kebutuhan dan memastikan sistem berjalan sesuai 
spesifikasi. 
4. Stakeholder lain, seperti karyawan bengkel yang menggunakan sistem secara langsung 
dalam proses pelayanan. 
1.3 Batasan Produk 
 Perangkat lunak yang dirancang adalah Sistem Informasi Bengkel Motor Fatia Kamal yang 
berfokus pada pengelolaan kegiatan operasional bengkel yang meliputi: 
1. Pendataan pelanggan dan kendaraan. 
2. Pencatatan layanan servis dan pembelian suku cadang. 
3. Pengelolaan stok barang dan data mekanik. 
 
 
4. Pembuatan laporan transaksi dan aktivitas servis. 
Sistem ini tidak mencakup fitur pembayaran online, integrasi dengan sistem pihak ketiga, atau 
sistem keuangan yang kompleks. Produk dikembangkan berbasis web agar dapat diakses melalui 
perangkat komputer atau laptop oleh admin bengkel. 
1.4 Definisi dan Istilah 
● SRS   : Software Requirements Specification 
     Dokumen spesifikasi kebutuhan perangkat lunak yang berisi deskripsi  
      lengkap mengenai fungsi, batasan, dan kebutuhan sistem. 
● IEEE  : Institute of Electrical and Electronics Engineers 
     Lembaga internasional yang menetapkan standar pengembangan dan  
     perancangan perangkat lunak dan perangkat keras. 
● IS   : Information System 
Aplikasi berbasis komputer yang digunakan untuk mengumpulkan, 
mengelola, dan menampilkan data agar mendukung kegiatan operasional dan 
pengambilan keputusan. 
● USER   : Pengguna 
Pihak yang menggunakan sistem, seperti admin bengkel, mekanik, atau 
pemilik bengkel. 
● DBMS  : Database Management System 
Kumpulan data yang tersusun secara sistematis untuk menyimpan informasi 
pelanggan, kendaraan, dan transaksi.

2. Deskripsi Keseluruhan 
2.1 Deskripsi Produk  
 Sistem Informasi Bengkel Motor Fatia Kamal merupakan aplikasi berbasis web yang 
dirancang untuk mempermudah pengelolaan operasional bengkel. Sistem ini mencakup pendataan 
pelanggan dan kendaraan, pengelolaan stok suku cadang, pencatatan layanan servis, serta pembuatan 
laporan transaksi dan aktivitas bengkel. Selain itu, sistem juga menyediakan fitur manajemen 
 
 
mekanik dan penjadwalan servis, sehingga mempermudah koordinasi antar staf bengkel. Sistem 
dirancang secara modular agar dapat dikembangkan di masa depan, termasuk kemungkinan integrasi 
dengan sistem keuangan atau pembayaran online. Dengan antarmuka yang s ederhana dan responsif, 
sistem ini dapat digunakan oleh admin bengkel, mekanik, maupun pelanggan secara efektif dan 
efisien. 
2.2 Fungsi Produk  
 Sistem Informasi Bengkel Motor Fatia Kamal dirancang untuk mendukung seluruh kegiatan 
operasional bengkel secara efisien dan terstruktur. Secara garis besar, sistem memiliki beberapa 
fungsi utama sebagai berikut: 
- Mencatat data pelanggan dan kendaraan secara lengkap dan terstruktur. 
- Mengelola stok suku cadang dan bahan habis pakai dengan akurat. 
- Mencatat layanan servis yang dilakukan oleh mekanik. 
- Membuat laporan transaksi dan aktivitas bengkel secara periodik.

dalam menjalankan fungsi operasional bengkel, mulai dari pencatat an transaksi hingga pengawasan 
kinerja. Oleh karena itu, pengelompokan karakteristik pengguna dilakukan untuk memastikan sistem 
dapat dirancang sesuai kemampuan, kebutuhan, dan hak akses masing-masing.

2.4 Lingkungan Operasi  
Sistem ini dirancang berbasis web dan beroperasi pada lingkungan sebagai berikut: 
● Platform: Aplikasi berbasis web menggunakan  browser modern (Google Chrome, 
Mozilla Firefox, Microsoft Edge). 
● Server: Apache atau Nginx dengan PHP ≥ 8.0 dan MySQL/MariaDB sebagai basis data. 
● Client: PC atau laptop dengan koneksi internet dan resolusi layar minimal 1366x768. 
● Sistem Operasi: Kompatibel dengan Windows, Linux, atau macOS. 
● Komponen Pendukung: 
○ Web server (XAMPP atau LAMP). 
○ Browser dengan JavaScript aktif. 
○ Framework CSS (TailwindCSS / Bootstrap). 
2.5 Batasan Desain dan Implementasi  
Beberapa batasan dalam desain dan implementasi sistem ini adalah: 
1. Teknologi: Sistem dikembangkan menggunakan PHP Native dengan database MySQL 
dan tampilan berbasis HTML, CSS, serta JavaScript. 
2. Aksesibilitas: Sistem hanya dapat diakses oleh pengguna terdaftar melalui jaringan lokal 
atau internet dengan login. 
3. Keamanan: Menggunakan hashing untuk penyimpanan kata sandi (password_hash). 
4. Performa: Sistem dirancang untuk menangani hingga ±1000 data pelanggan tanpa 
penurunan performa signifikan. 
5. Keterbatasan Hardware: Server minimal 2GB RAM dan 20GB storage. 
 
 
6. Standar Pemrograman: Mengacu pada prinsip Clean Code  dan struktur Model -View-
Controller (MVC) sederhana. 
7. Regulasi: Tidak terhubung dengan sistem eksternal atau pembayaran online demi 
menjaga privasi data pelanggan. 
2.6 Dokumentasi Pengguna 
Dokumentasi yang disediakan bersama sistem meliputi: 
1. User Manual (PDF): Panduan penggunaan sistem, mulai dari login hingga pengelolaan 
data dan pembuatan laporan. 
2. Panduan Instalasi: Langkah -langkah instalasi sistem di server lokal menggunakan 
XAMPP. 
3. Online Help: Petunjuk singkat di dalam aplikasi pada tiap form untuk membantu 
pengguna. 
4. Video Tutorial: Penjelasan cara penggunaan fitur utama (opsional).

3.1 Struktural 
3.1.1 Data Flow Diagram Level 0 
 
Diagram ini menggambarkan hubungan dan aliran data 8ar uke 8ni8em dengan entitas 
eksternal yang terlibat, yaitu admin bengkel dan pemilik bengkel. Admin bengkel berperan sebagai 
pengguna utama yang melakukan berbagai aktivitas operasional, seperti mencatat data sparepart 
masuk dan keluar, data biaya servis, jenis motor pelanggan, status atau progres pengerjaan, keluhan 
pelanggan, serta transaksi keuangan. Seluruh data tersebut dikirim ke sistem untuk diproses dan 
disimpan di dalam database sparepart. Selanjutnya, sistem akan mengelola dan menghasilkan 
berbagai laporan, s eperti laporan servis, laporan stok sparepart, dan laporan keuangan. Laporan -
laporan tersebut kemudian dapat diakses oleh pemilik bengkel sebagai bahan evaluasi dan 
pengambilan keputusan dalam pengelolaan usaha. Melalui aliran data ini, sistem membantu 
memastikan bahwa setiap informasi operasional bengkel tercatat, tersimpan, dan dapat digunakan 
secara efektif untuk mendukung kinerja bengkel secara keseluruhan. 

 
 
3.1.2 Data Flow Diagram Level 1 
DFD Level 1 ini menggambarkan rincian alur data pada sistem pengelolaan bengkel dengan 
empat proses utama yang saling terhubung. Proses dimulai dari Admin yang memasukkan form data 
service ke dalam proses 1.0 Kelola Data Service, yang bertugas mencatat dan memvalidasi informasi 
service pelanggan. Dari proses ini, sistem menghasilkan data service pelanggan sekaligus 
mengirimkan data permintaan sparepart  ke proses 2.0 Pengelolaan Sparepart. Proses pengelolaan 
sparepart kemudian memeriksa ketersediaan stok, mengeluarkan sparepart yang diperlukan, serta 
memperbarui data stok sparepart. Informasi tentang sparepart yang keluar dikirimkan kembali untuk 
melengkapi data pada proses pembayaran. Selanjutnya, data service pelanggan dikirim ke proses 3.0 
Kelola Pembayaran, yang bertugas menghitung biaya service, mencatat transaksi pembayaran, dan 
menghasilkan data pembayaran. Data pembayaran tersebut menjadi salah satu sumber utama untuk 
proses 4.0 Kelola Laporan, yang menggabungkan data pembayaran dan data service guna 
menghasilkan laporan operasional bengkel. Laporan ini kemudian diterima oleh Owner sebagai pihak 
yang membutuhkan informasi untuk pengawasan dan pengambilan keputusan. Secara keseluruhan, 

 
 
DFD Level 1 ini menunjukkan integrasi antara pengelolaan service, stok sparepart, pembayaran, dan 
pelaporan dalam satu alur sistem yang terstruktur dan saling terkait. 
3.1.3 Logical Level 
Logical Level pada sistem manajemen bengkel ini menggambarkan empat proses utama yang 
saling terhubung dengan tiga data store serta satu aktor, yaitu Admin Bengkel. Pada proses 
Pendaftaran Service, admin melakukan input data berupa keluhan pelanggan dan jenis motor, lalu 
sistem menyimpannya sebagai data service baru ke dalam D1_Service. Selanjutnya, proses 
Manajemen Inventaris mengelola seluruh aktivitas terkait stok sparepart, mulai dari memperbarui 
data sparepart masuk dan keluar hingga menyediakan inform asi kebutuhan sparepart berdasarkan 
jadwal service, yang kemudian disimpan dalam D2_Sparepart. Proses Servis dan Transaksi  
menangani pengerjaan servis, pencatatan penggunaan sparepart, status pekerjaan, biaya, serta data 
transaksi yang seluruhnya diperbarui ke dalam D3_Keuangan. Ketiga data store tersebut menjadi 
sumber informasi utama bagi proses Pelaporan, yang menghasilkan laporan service, laporan stok, dan 
laporan keuangan. Semua laporan ini kemudian diberikan kembali kepada Admin Bengkel sebagai 
bahan evaluasi dan pengambilan keputusan. Logical level ini menunjukkan alur data yang terstruktur 
dan hubungan antarproses yang mendukung operasional bengkel secara menyeluruh. 
3.1.4 Physical Level 

 
 
Physical Level pada 11ar uke 11ni menjelaskan bagaimana proses- proses dalam bengkel 
dijalankan secara nyata, termasuk interaksi langsung antara admin, sistem, dan penyimpanan data. 
Proses dimulai dari Input Data Service, di mana admin mengisi form pendaftaran service ke dalam 
sistem. Data  ini kemudian disimpan sebagai data service 11ar uke  dalam D1_Service dan secara 
otomatis diteruskan sebagai data service pelanggan ke proses berikutnya. Selanjutnya, proses Kelola 
Stok Sparepart bekerja untuk mengubah data stok berdasarkan penggunaan sparepart selama service. 
Sistem mengambil data stok lama dari D2_Sparepart, lalu memperbaruinya dan mengembalikan stok 
terbaru ke data store tersebut. Proses ini juga mengirim informasi sparepart yang digunakan ke proses 
pembayaran. Pada proses Pembayaran, sistem menerima data service pelanggan dan data sparep art 
yang digunakan untuk menghitung total biaya dan mencatat transaksi. Data pembayaran yang telah 
diproses disimpan ke dalam D3_Pembayaran sebagai data pembayaran baru. Seluruh informasi 
mengenai pembayaran dan service kemudian digunakan oleh proses Buat Laporan, yang 
menghasilkan laporan operasional bengkel. Laporan tersebut dikirimkan kepada Pemilik sebagai 
bahan evaluasi dan pengambilan keputusan. Secara keseluruhan, physical level ini menggambarkan 
implementasi nyata dari aliran data, pengolahan service, pengelolaan stok, transaksi pembayaran, dan 
penyusunan laporan dalam operasi bengkel.

3.1.6 OO-DFD 

 
 
OO-DFD ini menggambarkan aliran data pada sistem bengkel berbasis objek, di mana setiap 
proses utama terhubung dengan class-class yang menangani data tertentu. Proses dimulai dari Admin 
yang mengirimkan form data service ke proses 1.0 Kelola Data Service . Pada proses ini, sistem 
menjalankan fungsi dari objek Transaksi  untuk membuat data service awal, yang kemudian 
menghasilkan data service pelanggan serta data permintaan sparepart . Data permintaan tersebut 
diteruskan ke proses 2.0 Pengelolaan Sparepart, yang berinteraksi langsung dengan objek Sparepart  
untuk memeriksa stok, mengurangi sparepart yang digunakan, serta memperbarui jumlah stok melalui 
metode seperti UpdateStok() dan editSparepart(). Output dari proses kedua berupa data sparepart keluar, 
yang dikirim ke proses berikutnya. 
Selanjutnya, data service pelanggan dan data sparepart keluar digunakan oleh proses 3.0 Kelola 
Pembayaran, di mana objek Transaksi  dipanggil untuk menghitung total biaya melalui method 
hitungTotal(), melakukan pencatatan sparepart yang digunakan, dan menyimpan transaksi 
menggunakan simpanTransaksi(). Outputnya berupa data pembayaran, yang menjadi input bagi proses 
4.0 Kelola Laporan. Pada tahap ini, sistem memanfaatkan objek Laporan Keuangan untuk 
menghitung total pemasukan dan menghasilkan laporan menggunakan method buatLaporan(). Laporan 
yang dihasilkan kemudian disampaikan kepada Owner, yang dapat meninjau laporan tersebut sebagai 
bentuk evaluasi operasional bengkel. 
Secara keseluruhan, OO-DFD ini menunjukkan interaksi terstruktur antara proses dan objek, di 
mana setiap aliran data tidak hanya melewati proses fungsional tetapi juga berhubungan langsung 
dengan class yang bertanggung jawab terhadap pengelolaan data. Diagram ini menegaskan 
bagaimana konsep berorientasi objek diterapkan dalam aliran data, dengan setiap proses 
memanfaatkan atribut dan metode yang ada dalam class untuk memastikan integritas data, alur kerja 
yang jelas, serta konsistensi dalam pengolahan infor masi pada sistem bengkel. Jika kamu mau, aku 
bisa buatkan versi ringkas atau versi sangat formal untuk dimasukkan ke laporan.

3.2.5 Class Diagram 
 
Class Diagram pada sistem informasi bengkel ini menggambarkan hubungan antar bagian 
utama yang berperan dalam proses operasional, mulai dari transaksi service, pengelolaan sparepart, 
hingga pembuatan laporan. Diagram ini terdiri dari lima kelas utama, yaitu Admin, Transaksi, 
Sparepart, Laporan Keuangan, dan Owner , yang masing -masing memiliki atribut serta fungsi 
(method) sesuai perannya. Kelas Admin  mewakili pengguna sistem dengan hak akses untuk 
melakukan login, mengelola transaksi, memperbarui stok sparepart, serta melihat data transaksi. 
Admin menjadi pusat aktivitas karena setiap proses operasional bengkel dilakukan oleh admin yang 
tercatat dalam sistem. Kelas Transaksi berisi data terkait pelayanan service, seperti tanggal transaksi, 
nama pelanggan, jenis motor, total biaya, status, hingga catatan keluhan. Kelas ini memiliki hubungan 
langsung dengan Admin dan Sparepart karena transaksi service dapat melibatkan perubahan stok 

 
 
sparepart dan dicatat oleh admin. Selain itu, setiap transaksi juga berkontribusi terhadap laporan 
keuangan yang tercatat pada kelas Laporan Keuangan. 
Kelas Sparepart menyimpan informasi terkait barang yang digunakan dalam proses service, 
seperti nama sparepart, harga beli, harga jual, dan jumlah stok. Relasinya dengan kelas Transaksi 
menunjukkan bahwa setiap service dapat mengurangi stok sparepart, sehingga kedua kelas ini saling 
memengaruhi. Kelas Laporan Keuangan bertugas mencatat pemasukan dan pengeluaran berdasarkan 
transaksi yang terjadi. Setiap transaksi yang tersimpan akan berhubungan dengan laporan keuangan 
untuk menghasilkan total pendapatan dan total biaya operasional. Laporan ini kemudian dapat diakses 
oleh kelas Owner, yang berperan sebagai pemilik bengkel. Kelas Owner memiliki kemampuan untuk 
login dan melihat laporan keuangan maupun laporan transaksi yang dibuat sistem, namun tidak dapat 
mengubah data operasional lainnya. 
Secara keseluruhan, Class Diagram ini menunjukkan bagaimana data transaksi, sparepart, dan 
laporan saling terhubung melalui peran admin sebagai pengelola utama. Struktur hubungan antar 
kelas dirancang agar sistem dapat berjalan secara terintegrasi, sehingg a memudahkan proses 
pencatatan transaksi, monitoring stok sparepart, serta penyusunan laporan keuangan yang dibutuhkan 
oleh pemilik bengkel. 
3.2.6 Entity Relationship Diagram ERD 
ERD ini menggambarkan sistem informasi bengkel motor yang melibatkan beberapa  entitas 
utama seperti Transaksi, Admin, Owner, Sparepart, dan Laporan Keuangan. Tujuannya adalah untuk 
mengelola proses bisnis bengkel mulai dari pencatatan transaksi, pengelolaan data sparepart, hingga 
pembuatan dan pengawasan laporan keuangan. 
 
 
 
Entity Relationship Diagram (ERD) merupakan model konseptual yang digunakan untuk 
menggambarkan hubungan antar entitas (tabel) dalam sebuah sistem basis data. ERD membantu 
perancang sistem memahami struktur data serta interaksi antar komponen yang ada di d alam sistem. 
Dalam konteks Sistem Informasi Bengkel Motor Fatia Kamal, ERD digunakan untuk memetakan data 
utama yang dibutuhkan dalam pengelolaan operasional bengkel agar informasi dapat tersimpan 
dengan baik, saling terhubung, dan mudah diolah. 
Pada ERD di atas, terdapat beberapa entitas utama, yaitu Admin, Owner, Transaksi, Laporan 
Keuangan, dan Sparepart. Entitas Admin memiliki peran penting dalam sistem karena bertugas untuk 
mencatat data transaksi pelanggan, mengelola data sparepart, dan membuat laporan keuangan. Setiap 
transaksi yang dilakukan pelanggan, seperti servis atau pembelian suku cadang, akan dicatat oleh 
admin ke dalam entitas Transaksi, yang berisi informasi seperti tanggal transaksi, nama pelanggan, 
jenis motor, total biaya, status, dan catatan keluhan. 
Selanjutnya, entitas Laporan Keuangan menampung hasil pengolahan data transaksi yang 
dibuat oleh admin. Laporan ini berisi tanggal laporan, total uang masuk, dan uang keluar, yang 

 
 
menjadi dasar pemantauan keuangan bengkel. Entitas Owner memiliki hubungan langsung dengan 
laporan keuangan dan transaksi, karena pemilik bengkel berperan dalam mengawasi seluruh aktivitas 
keuangan dan melihat laporan yang dihasilkan oleh sistem. 
Selain itu, entitas Sparepart menyimpan data terkait stok, harga beli, dan harga jual suku 
cadang. Admin bertanggung jawab untuk mengelola entitas ini agar ketersediaan barang selalu 
tercatat dengan akurat. Hubungan antar entitas pada ERD ini menunjukkan keterkaitan fungsional 
antara kegiatan operasional bengkel dengan sistem informasi yang dibangun. Dengan adanya model 
data seperti ini, sistem dapat membantu pengelolaan bengkel menjadi lebih efisien, terstruktur, dan 
meminimalisir kesalahan pencatatan manual.

4.3 Hardware Interface 
Sistem memerlukan perangkat keras yang mendukung operasi berbasis web dengan 
spesifikasi minimum sebagai berikut: 
● Server: 
○ Prosesor minimal Intel Core i3 atau setara. 
○ RAM minimal 2 GB. 
○ Kapasitas penyimpanan minimal 20 GB. 
○ Mendukung sistem operasi Windo ws, Linux, atau macOS. 
● Client: 
○ Komputer atau laptop dengan koneksi internet. 
○ Resolusi layar minimal 1366x768. 
○ Browser modern (Chrome, Firefox, Edge). 
● Perangkat Tambahan: 
○ Printer (opsional) untuk mencetak laporan transaksi dan data servis. 
4.4 Software Interface 
Sistem berinteraksi dengan beberapa komponen perangkat lunak lainnya untuk memastikan 
fungsionalitas berjalan dengan baik. 
● Database Management System: MySQL atau MariaDB digunakan untuk menyimpan 
seluruh data pelanggan, kendaraan, stok, dan transaksi. 
● Web Server: Apache (melalui XAMPP/LAMP) digunakan sebagai server utama. 
● Bahasa Pemrograman: PHP 8.x untuk logika backend, HTML5, CSS3 
(TailwindCSS/Bootstrap), dan JavaScript untuk antarmuka pengguna. 
● Sistem Operasi: Kompatibel dengan Windows, Linux, dan macOS. 
● Library Pendukung: 
○ PHPMyAdmin untuk administrasi database. 
○ Chart.js untuk menampilkan laporan dalam bentuk grafik. 
● Browser Kompatibilitas: Mendukung Chrome, Firefox, Edge, dan Safari versi terbaru. 
4.5 Communication Interface 
Sistem beroperasi menggunakan protokol komunikasi berbasis web dengan standar keamanan 
dasar. 
● Protokol Komunikasi: HTTP/HTTPS untuk pertukaran data antara client dan server. 
 
 
● Format Data: HTML dan JSON digunakan untuk pertukaran data antara frontend dan 
backend. 
● Keamanan Komunikasi: 
○ Penggunaan HTTPS disarankan untuk mengenkripsi data. 
○ Validasi input pada sisi client dan server untuk mencegah serangan injeksi. 
● Koneksi Jaringan: Sistem dapat diakses melalui jaringan lokal (LAN) maupun internet. 
● Kecepatan Akses: Disarankan minimal 1 Mbps untuk performa optimal. 
Port yang Digunakan: Port 80 untuk HTTP dan port 443 untuk HTTPS. 
5 Non Functional Requirement 
ID Parameter Kebutuhan 
NFR-01 Availability Sistem harus dapat diakses setiap hari selama jam operasional 
bengkel (08.00– 20.00) tanpa gangguan, dengan tingkat 
ketersediaan minimal 95%. 
NFR-02 Reliability Sistem harus mampu menyimpan dan menampilkan data 
pelanggan, transaksi, serta laporan dengan akurat tanpa 
kehilangan data. Toleransi kesalahan maksimal 1% dari total 
transaksi bulanan. 
NFR-03 Ergonomy Antarmuka sistem harus sederhana, mudah dipahami, dan

NFR-04 Portability Sistem berbasis web harus dapat berjalan pada berbagai sistem 
operasi seperti Windows, Linux, dan macOS, serta dapat diakses 
melalui browser Chrome, Firefox, dan Edge tanpa instalasi 
tambahan. 
NFR-05 Memory Aplikasi dan database sistem harus mampu berjalan optimal pada 
server dengan RAM minimal 2 GB dan ruang penyimpanan 20 
GB tanpa menimbulkan keterlambatan signifikan. 
NFR-06 Response time Sistem harus mampu menampilkan halaman utama atau hasil 
pencarian data dalam waktu maksimal 3 detik pada koneksi 
internet 1 Mbps.

NFR-07 Safety Tidak berlaku (N/A) karena sistem ini tidak berhubungan

NFR-08 Security Sistem harus menggunakan autentikasi login dengan username 
dan password terenkripsi (menggunakan password_hash). 
Setiap pengguna memiliki hak akses berbeda sesuai peran 
(Admin, Mekanik, Pemilik). Selain itu, sistem harus mencegah 
SQL Injection dan validasi input di sisi server maupun client.
